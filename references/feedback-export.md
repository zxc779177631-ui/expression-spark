# 试用反馈导出

目标是让维护者同时看到“积累出了什么”和“交互为什么有效或失效”，而不是收到一份混有无关内容、凭证和工具日志的完整聊天记录。

## 导出内容

反馈包包含：

1. `expression-spark-outcomes.md`：由 `library.py feedback` 生成的成果快照。
2. `interaction-feedback.md`：Agent 根据本轮试用整理的交互反馈。
3. `runtime-diagnostics.md`：仅在出现权限、路径、命令或运行时错误时创建。

默认不包含完整聊天记录，不自动上传，不包含无关任务对话。

## 第一步：导出成果快照

默认只导出统计和资产索引：

```bash
python3 "$LIBRARY_TOOL" feedback \
  --library "<资产库路径>" \
  --output "<反馈目录>/expression-spark-outcomes.md"
```

用户明确同意把已确认语料交给维护者评估时：

```bash
python3 "$LIBRARY_TOOL" feedback \
  --library "<资产库路径>" \
  --include-content \
  --session-limit 12 \
  --output "<反馈目录>/expression-spark-outcomes.md"
```

`--include-content` 仍只包含资产库中经确认保存的内容，不包含完整聊天记录。命令会遮盖常见密钥格式，但 Agent 仍要人工检查。

## 第二步：整理交互反馈

创建 `interaction-feedback.md`，使用以下结构：

```markdown
# Expression Spark 交互反馈

## 试用范围
- 日期：
- 使用入口：
- 大致会话数：
- Skill 版本或 commit：

## 用户明确确认的个人偏好
- 仅记录这个用户更有感觉的互动方式，不直接泛化给所有用户。

## 可泛化的产品发现
- 说明证据来自什么现象，以及还需要怎样验证。

## 有效交互样本
### 样本 1
- Agent 当时怎么问：
- 用户为什么被聊开：
- 产生了什么成果：

## 无效交互或用户纠正
### 样本 1
- Agent 当时怎么问/做：
- 用户如何纠正：
- 应调整什么：

## 成果质量观察
- 哪些短会话后来组成了同一主题：
- 哪些原话形成了选题：
- 哪些会话仍只有原话、值得周期性归并：
- 哪些画像像本人，哪些不准确：

## 待验证假设
- 下一轮只改变一个变量，说明如何判断有效。

## 隐私检查
- [ ] 未包含密码、App Secret、API Key、Token
- [ ] 未包含与试用无关的聊天
- [ ] 精确引用仅来自与 Expression Spark 相关的必要片段
- [ ] 用户已同意分享包含的已确认语料
```

每类只选择 2–5 个高信息量样本。保留必要的精确问句和用户纠正，不要复制整段会话。

## 第三步：分离运行环境问题

权限、目录归属、运行时 Skill 扫描路径、平台工具失败和服务器限制，写入 `runtime-diagnostics.md`。不要在没有跨环境复现前，把它们直接判断为 Skill 设计缺陷。

记录：

- 运行时与操作系统。
- Skill 实际安装路径。
- 资产库实际路径。
- 执行命令。
- 精简后的错误信息。
- 是否在普通电脑或另一运行时复现。

## 给 Hermes 的直接要求

用户可以直接说：

> 请按 expression-spark 的 `references/feedback-export.md` 导出本轮试用反馈。先生成不含完整聊天记录的成果快照，再整理个人偏好、可泛化发现、有效/无效交互样本和运行环境错误。不要导出无关对话，检查并遮盖所有密钥。把文件打包后告诉我路径。

若用户希望维护者直接评估已确认语料，再补一句：

> 我同意反馈包包含 expression-spark 资产库里已经确认保存的语料，但仍不要包含完整聊天记录或无关内容。
