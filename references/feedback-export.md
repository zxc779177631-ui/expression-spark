# 试用反馈导出

目标是让维护者同时看到“积累出了什么”和“交互为什么有效或失效”，而不是收到一份混有无关内容、凭证和工具日志的完整聊天记录。

## 导出内容

反馈包包含：

1. `expression-spark-outcomes.md`：由 `library.py feedback` 生成的成果快照。
2. `interaction-feedback.md`：Agent 根据本轮试用整理的交互反馈。
3. `runtime-diagnostics.md`：仅在出现权限、路径、命令或运行时错误时创建。

默认不包含完整聊天记录，不自动上传，不包含无关任务对话。

## 证据质量门禁

反馈文件的目标不是“写一份完整总结”，而是提供可以复核的真实证据。生成前先运行：

```bash
python3 "$LIBRARY_TOOL" version
python3 "$LIBRARY_TOOL" feedback --help
python3 "$LIBRARY_TOOL" status --library "<资产库路径>"
```

如果 `version` 或 `feedback` 不存在，说明正在使用旧版或错误安装的 Skill。先记录运行环境问题，不要声称已按新版反馈流程导出。

遵守以下规则：

- 每个有效/无效交互样本必须来自实际试用对话，并标注来源会话 ID、日期或时间；不能复制 `SKILL.md` 里的示例问题。
- “形成了选题”“成为 confirmed 信号”等成果判断必须给出对应 topic/signal ID，并与 `status` 和成果快照一致。
- 找不到证据时写“尚未验证”，不要根据印象补全。
- 不把路径、权限、CLI 参数或部署限制写成用户的个人表达偏好。
- 区分三类信息：
  - **个人表达偏好**：什么互动更容易让该用户愿意表达。
  - **产品工作流偏好**：保存、审阅、模式展示、导出等产品交互偏好。
  - **运行环境约束**：路径、权限、安装方式和工具错误。
- 用户纠正 Agent 的事实错误，不自动等同于用户喜欢“纠错式表达”；只有用户明确确认或跨会话重复出现时，才记为个人触发器。

## 第一步：导出成果快照

默认只导出统计和资产索引：

```bash
python3 "$LIBRARY_TOOL" feedback \
  --library "<资产库路径>" \
  --output "<反馈目录>/expression-spark-outcomes.md"
```

用户明确同意把已确认语料交给维护者评估时：

```bash
python3 "$LIBRARY_TOOL" feedback \
  --library "<资产库路径>" \
  --include-content \
  --session-limit 12 \
  --output "<反馈目录>/expression-spark-outcomes.md"
```

`--include-content` 仍只包含资产库中经确认保存的内容，不包含完整聊天记录。命令会遮盖常见密钥格式，但 Agent 仍要人工检查。

## 第二步：整理交互反馈

创建 `interaction-feedback.md`，使用以下结构：

```markdown
# Expression Spark 交互反馈

## 试用范围
- 日期：
- 使用入口：
- 大致会话数：
- Skill 版本或 commit：

## 用户明确确认的个人偏好
- 仅记录这个用户更有感觉的互动方式，不直接泛化给所有用户。
- 证据：

## 产品工作流偏好
- 保存、审阅、模式展示或反馈导出的偏好，不写入个人表达画像。
- 证据：

## 可泛化的产品发现
- 说明证据来自什么现象，以及还需要怎样验证。

## 有效交互样本
### 样本 1
- 来源会话/时间：
- Agent 当时怎么问：
- 用户实际怎么答：
- 用户为什么被聊开：
- 产生了什么成果：
- 关联资产 ID：

## 无效交互或用户纠正
### 样本 1
- 来源会话/时间：
- Agent 当时怎么问/做：
- 用户如何纠正：
- 应调整什么：

## 成果质量观察
- 哪些短会话后来组成了同一主题：
- 哪些原话形成了选题：
- 哪些会话仍只有原话、值得周期性归并：
- 哪些画像像本人，哪些不准确：

## 待验证假设
- 下一轮只改变一个变量，说明如何判断有效。

## 隐私检查
- [ ] 未包含密码、App Secret、API Key、Token
- [ ] 未包含与试用无关的聊天
- [ ] 精确引用仅来自与 Expression Spark 相关的必要片段
- [ ] 用户已同意分享包含的已确认语料

## 一致性检查
- [ ] 所有“confirmed/recurring”描述与成果快照状态一致
- [ ] 所有声称形成的选题或信号都有资产 ID
- [ ] 有效问题来自真实对话，而不是 Skill 示例
- [ ] 环境问题没有混入个人表达偏好
```

每类只选择 2–5 个高信息量样本。保留必要的精确问句和用户纠正，不要复制整段会话。

## 第三步：分离运行环境问题

权限、目录归属、运行时 Skill 扫描路径、平台工具失败和服务器限制，写入 `runtime-diagnostics.md`。不要在没有跨环境复现前，把它们直接判断为 Skill 设计缺陷。

记录：

- 运行时与操作系统。
- Skill 实际安装路径。
- 资产库实际路径。
- 执行命令。
- 精简后的错误信息。
- 是否在普通电脑或另一运行时复现。

用户可以直接说：

若用户希望维护者直接评估已确认语料，再补一句：

> 我同意反馈包包含 expression-spark 资产库里已经确认保存的语料，但仍不要包含完整聊天记录或无关内容。
